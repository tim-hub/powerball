## Context

The `openspec-rewrite` plugin contains 9 skills — 8 generated by the openspec CLI (with extra frontmatter keys like `license`, `compatibility`, `metadata`) and 1 hand-written (`setup-openspec`). The hand-written one already follows repo conventions. The 8 generated ones need normalization.

The repo convention (established in `core/` skills like `commit`, `tag`, `skill-review`) is:
- Frontmatter order: `name` → `description` → `user-invocable` → `model` → others
- `argument-hint` declared when a skill accepts arguments
- `model` always specified
- Imperative writing style, no second person
- Lean body with `references/` for supplementary content

## Goals / Non-Goals

**Goals:**
- Normalize all 8 generated skills to match repo frontmatter conventions
- Add missing `argument-hint` and `model` keys
- Extract duplicated "Artifact Creation Guidelines" to a shared location
- Simplify `openspec-explore` by extracting examples and fixing second-person
- Simplify `openspec-verify-change` by extracting output templates and heuristics

**Non-Goals:**
- Removing the extra `license`, `compatibility`, `metadata` keys (per guardrails: never remove existing frontmatter keys)
- Fixing trigger phrase collisions (apply vs continue, new vs propose) — that's a separate semantic change
- Changing skill behavior or logic
- Touching `setup-openspec` (already correct)

## Decisions

### Decision 1: Keep non-standard frontmatter keys, reorder only

The generated skills have `license`, `compatibility`, `metadata` keys that don't exist in core skills. Per the skill-review guardrails ("never remove frontmatter keys"), these will be preserved but moved after the standard keys.

**Final order:** `name` → `description` → `argument-hint` → `user-invocable` → `model` → `allowed-tools` → `agent` → `context` → `disable-model-invocation` → `license` → `compatibility` → `metadata`

**Alternative:** Strip the non-standard keys entirely. Rejected because the guardrails explicitly prohibit key removal, and these keys may be read by the openspec system.

### Decision 2: Shared references via plugin-level `references/` directory

The duplicated "Artifact Creation Guidelines" block appears in both `openspec-propose` and `openspec-continue-change`. Extract it to `openspec-rewrite/references/artifact-guidelines.md` and reference it from both skills.

**Alternative:** Put it in each skill's own `references/` directory. Rejected because the content is identical and must stay in sync — a single source of truth prevents drift.

### Decision 3: Model assignment

Skills that do lightweight orchestration (archive, new, sync, continue) get `model: haiku`. Skills that do heavy reasoning (apply, verify, explore) keep no model specified or get a higher-tier model. `openspec-propose` already has `model: opus` — keep it.

| Skill | Model |
|-------|-------|
| openspec-apply-change | (no model — needs full reasoning for code implementation) |
| openspec-archive-change | haiku (already set) |
| openspec-continue-change | (no model — creates artifacts requiring reasoning) |
| openspec-explore | (no model — deep thinking mode) |
| openspec-new-change | haiku (scaffolding only) |
| openspec-propose | opus (already set) |
| openspec-sync-specs | haiku (mechanical merge) |
| openspec-verify-change | (no model — analysis requires reasoning) |

After review: only `openspec-new-change` and `openspec-sync-specs` need `model: haiku` added. The others intentionally omit it to allow full-capability model selection.

## Risks / Trade-offs

- **Risk:** Reordering frontmatter could break openspec CLI parsing if it depends on key order → **Mitigation:** YAML is order-independent; this is safe.
- **Risk:** Extracting content to `references/` could make skills harder to understand in isolation → **Mitigation:** Keep a one-line summary in the body with a pointer to the reference file.
- **Risk:** Adding `model: haiku` to wrong skills could degrade quality → **Mitigation:** Only add to clearly mechanical/lightweight skills (new-change, sync-specs). Leave reasoning-heavy skills without a model constraint.
